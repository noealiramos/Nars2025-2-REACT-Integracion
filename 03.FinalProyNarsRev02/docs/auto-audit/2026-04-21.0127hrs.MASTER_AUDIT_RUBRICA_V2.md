## 6. PLAN PARA LLEGAR A 130

### Alto impacto

- Publicar y validar deploy real:
  - Backend en Render con URL publica y smoke test de `/` y `/api/products`
  - Frontend en Vercel apuntando a `VITE_API_URL` publica

### Medio impacto

- Convertir mas rutas pesadas a `lazy`, al menos admin, orders, login y register
- Revalidar sesion inicial con `GET /api/users/me` o refresh controlado antes de marcar auth como restaurada
- Agregar flujo de carrito invitado o documentar y evaluar si la rubrica acepta carrito solo autenticado

### Bajo impacto

- Crear dashboard admin mas explicito con entrypoint unico
- Generar smoke test automatizado de produccion para evitar que el deploy vuelva a quedar sin evidencia

## 7. RIESGOS

- Seguridad:
  - La sesion inicial basada en `localStorage` sin revalidacion puede dejar UI en estado optimista si el token ya no es valido.
- UX:
  - Los 2 tests fallidos de checkout sugieren fragilidad en el manejo de expiracion/submit del formulario.
  - El carrito condicionado a auth puede sentirse restrictivo para visitante nuevo.
- Arquitectura:
  - Estado global fragmentado entre contexts con `useState` y un `useReducer` limitado al UI feedback.
  - Lazy loading parcial deja parte del bundle inicial innecesariamente grande.
