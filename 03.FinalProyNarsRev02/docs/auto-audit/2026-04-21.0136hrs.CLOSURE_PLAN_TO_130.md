## 2. Diagnostico por gap

### 2.1 Checkout tests fallidos

- Archivo afectado: `ecommerce-app-Nars/src/pages/__tests__/CheckoutPage.test.jsx`
- Fallos observados en corrida previa:
  - `normaliza el vencimiento numerico y continua el submit`
  - `bloquea submit si el vencimiento 1/26 no cumple MM/YY`
- Diagnostico tecnico:
  - El helper de test `fillCheckoutForm()` usa `user.type()` para cargar todos los campos del checkout, incluyendo `input-cardExpiry`.
  - `CheckoutPage.jsx` normaliza el vencimiento en cada `onChange` con `normalizeExpiryDateInput()`.
  - `paymentExpiry.js` solo acepta `MM/YY` exacto via `EXPIRY_DATE_REGEX`.
  - La combinacion `user.type()` + formulario grande + validaciones asincronas probablemente hace que esos tests excedan el timeout por defecto de 5s, no necesariamente por bug funcional del componente.
  - La evidencia fuerte es que Cypress del checkout si paso completamente (`checkoutReuse`, `checkoutErrors`, `cart`, `goldenPath`).
- Hipotesis de causa raiz:
  - Fragilidad del test, no necesariamente fallo real de produccion.
  - Posible espera insuficiente o estrategia de tipeo lenta para un formulario con varias transiciones y llamadas async.

### 2.2 Context + useReducer parcial

- Estado actual:
  - `ecommerce-app-Nars/src/contexts/UIContext.jsx` ya usa `useReducer`.
  - `ecommerce-app-Nars/src/contexts/CartContext.jsx` sigue con varios `useState` (`items`, `cartId`, `isLoading`, `error`).
- Diagnostico tecnico:
  - El carrito es el mejor candidato para migrar a `useReducer` porque concentra estado de negocio, transiciones claras y side effects controlables.
  - `CartContext` ya tiene una forma de estado bastante definida, por lo que la migracion puede ser de bajo riesgo si se conserva el API publico del contexto.

### 2.3 Lazy loading parcial

- Estado actual en `ecommerce-app-Nars/src/App.jsx`:
  - Lazy: `HomePage`, `ProductDetailPage`, `CheckoutPage`, `ProfilePage`
  - Eager: `CartPage`, `ConfirmationPage`, `OrdersPage`, `OrderDetailPage`, `AdminProductsPage`, `AdminCategoriesPage`, `WishlistPage`, `LoginPage`, `RegisterPage`, `NotFoundPage`
- Diagnostico tecnico:
  - Hay oportunidad clara de ganar el punto faltante sin tocar contratos.
  - Las paginas mas pesadas y menos frecuentes son admin, orders, auth y wishlist; moverlas a lazy es de bajo riesgo.

### 2.4 Auth bootstrap sin revalidacion backend

- Estado actual:
  - `ecommerce-app-Nars/src/contexts/AuthContext.jsx` restaura sesion leyendo `getCurrentUser()` desde `localStorage`.
  - `ecommerce-app-Nars/src/services/authService.js` persiste tokens y user, pero no ofrece `getMe()` o `bootstrapSession()` con revalidacion.
  - `ecommerce-app-Nars/src/api/authApi.js` no expone `me`, aunque el backend si tiene `GET /api/users/me`.
- Diagnostico tecnico:
  - La UI puede asumir sesion valida antes de confirmar con backend.
  - Esto afecta robustez de auth y tambien la coherencia con `PrivateRoute`, `AdminRoute` y `GuestOnlyRoute`.
- Mejora de bajo riesgo:
  - Agregar un bootstrap asincrono que use token persistido + `GET /api/users/me`.
  - Si falla, limpiar sesion; si funciona, hidratar user y auth.

### 2.5 Guest cart no integral

- Estado actual:
  - `CartContext.jsx` bloquea explicitamente add/remove/update/clear en guest mode.
  - El proyecto actual esta diseñado para compra autenticada.
  - Cypress `productAccess.cy.js` incluso valida que un usuario no autenticado no use CTA de carrito para comprar.
- Diagnostico tecnico:
  - Implementar guest cart integral ahora implica definir persistencia local, merge al login y revisar expectativas de checkout.
  - Eso toca UX, reglas de negocio y compatibilidad con E2E ya verdes.
- Conclusion preliminar:
  - Es un cambio de mayor riesgo relativo al puntaje.
  - Conviene primero documentar formalmente la decision y solo implementar una version minima si no rompe el cierre actual.

### 2.6 Deploy funcional verificable

- Estado actual:
  - Existe `render.yaml`.
  - Existe `ecommerce-app-Nars/vercel.json`.
  - No hay evidencia verificable de URL publica activa en la auditoria vigente.
- Diagnostico tecnico:
  - El gap de mayor impacto es documental/operacional, no tanto de codigo.
  - Para recuperar esos 5 puntos se necesita evidencia verificable: URL publica, smoke test y logs o capturas de respuesta real.
- Dependencias externas:
  - Variables reales de entorno, cuentas Render/Vercel, y despliegue efectivo.
