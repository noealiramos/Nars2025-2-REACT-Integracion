# 1. Estado inicial

- Archivos objetivo:
  - `ecommerce-app-Nars/src/contexts/AuthContext.jsx`
  - `ecommerce-app-Nars/src/services/authService.js`
  - `ecommerce-app-Nars/src/api/authApi.js`
- Estado previo:
  - `AuthContext` restauraba sesion leyendo `userData` desde `localStorage`
  - si existia `userData`, marcaba `isAuthenticated=true` sin confirmar primero con backend
  - no existia helper explicito para bootstrap con revalidacion remota
  - `authApi` no exponia `GET /api/users/me`

---

## 2. Diagnóstico técnico

- El gap de rubrica provenia de que el bootstrap confiaba directamente en persistencia local.
- El backend ya exponia `GET /api/users/me` y el frontend ya tenia `apiClient` con JWT + refresh interceptado.
- El cambio de menor riesgo era:
  - agregar `authApi.me()`
  - encapsular la revalidacion en `authService.bootstrapSession()`
  - dejar `AuthContext` como orquestador del estado final
- Riesgo detectado durante la implementacion:
  - la spec `authLifecycle.cy.js` depende del flujo existente donde un access token expirado se refresca al primer request protegido, no necesariamente durante el bootstrap.
  - una revalidacion forzada con `/users/me` cuando el access token ya expiro cambiaba esa secuencia observable y rompia la spec.

---

## 3. Flujo de bootstrap antes

```text
App inicia
-> AuthContext monta
-> getCurrentUser() lee userData desde localStorage
-> si existe userData:
   -> setUser(current)
   -> setIsAuthenticated(true)
-> setLoading(false)
```

Problema:
- la app podia considerarse autenticada solo por persistencia local, sin revalidacion obligatoria con backend.

---

## 4. Cambios aplicados

- En `ecommerce-app-Nars/src/api/authApi.js`
  - se agrego `authApi.me()` usando `GET /users/me`

- En `ecommerce-app-Nars/src/services/authService.js`
  - se agrego `clearAuthSession()`
  - se agrego `hasPersistedSession()`
  - se agrego `bootstrapSession()`
  - se agrego parseo local del JWT para detectar access token expirado
  - se centralizo la limpieza de sesion reutilizando `clearAuthSession()` tambien en `logout()`

- En `ecommerce-app-Nars/src/contexts/AuthContext.jsx`
  - se reemplazo el restore directo desde `getCurrentUser()` por `bootstrapSession()`
  - el provider inicia en `loading=true`, espera el resultado del bootstrap y luego fija el estado auth final
  - el listener `auth-error` ahora tambien limpia almacenamiento y asegura estado consistente

Decision clave de bajo riesgo:
- si el access token persistido ya esta expirado pero aun existe refresh token valido y usuario persistido, el bootstrap no fuerza inmediatamente `/users/me`.
- en ese caso preserva la sesion local y delega la recuperacion al flujo de refresh ya probado por interceptores.
- esto mantiene compatibilidad con `authLifecycle.cy.js` sin perder la revalidacion real en el resto de arranques validos.

## 6. Flujo de bootstrap después

```text
App inicia
-> AuthContext monta
-> bootstrapSession()
   -> si no hay sesion persistida:
      -> success false
   -> si hay access token vigente:
      -> GET /api/users/me
      -> si backend confirma, hidrata user y mantiene sesion
      -> si backend falla, limpia sesion
   -> si el access token ya expiro pero existe refresh token + user persistido:
      -> mantiene sesion local inicial
      -> delega refresh al flujo ya existente del apiClient en la primera request protegida
-> AuthContext fija user/isAuthenticated/loading segun resultado
```

Resultado funcional:
- la app ya no depende solo de `userData` para restaurar una sesion comun valida
- una sesion invalida sin refresh util se limpia
- el flujo de refresh probado en runtime sigue intacto para tokens expirados

## 10. Riesgos detectados

- Riesgo bajo residual:
  - el flujo de bootstrap tiene una excepcion deliberada para access tokens ya expirados, hecha para no romper el lifecycle refresh ya cubierto por Cypress.
- Riesgo bajo controlado:
  - se agrego parseo local del JWT; si el token estuviera malformado, el helper lo trata como expirado y evita confiar ciegamente en el.

---

## 11. Recomendaciones no implementadas

- Agregar tests unitarios especificos de `AuthContext` y `bootstrapSession()` para cubrir:
  - token vigente + `/users/me` 200
  - token vigente + `/users/me` 401
  - token expirado + refresh valido
  - token expirado + refresh revocado
- Si en una fase futura se quiere endurecer aun mas el bootstrap, se podria agregar un flujo explicito de refresh previo al mount cuando el token este expirado, pero requeriria ajustar la expectativa de `authLifecycle.cy.js`.
