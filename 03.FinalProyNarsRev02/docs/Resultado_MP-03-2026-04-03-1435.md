# Resultado MP-03 - 2026-04-03 1435

## 1. Resumen ejecutivo

MP-03 se ejecuto en modo controlado, incremental y sin tocar backend.

- Se endurecio el CTA de agregado al carrito en catalogo y detalle.
- Se unifico la politica visible y funcional para `auth + stock`.
- Se agrego cobertura unitaria/component para `ProductCard` y `ProductDetailPage`.
- Se agrego E2E focal real `productAccess.cy.js`.
- Se detecto una regresion real en `cart.cy.js` porque el spec seguia asumiendo que un usuario anonimo podia agregar desde catalogo; se corrigio el spec para alinearlo con la UX obligatoria de MP-03 y se revalido.

## 2. Objetivo y alcance

Objetivo ejecutado:

- endurecer la logica frontend para agregar productos al carrito desde catalogo y detalle
- aplicar reglas explicitas de autenticacion y stock disponible
- mantener contratos existentes de `auth`, `cart` y `checkout`

Fuera de alcance y no tocado:

- backend
- endpoints y payloads API
- checkout flow estructural
- `orderApi`, `cartApi`
- React Query
- lazy loading
- admin
- GuestOnly
- MP-04 o fases posteriores

## 3. Reglas UX definidas

Reglas implementadas igual en `ProductCard` y `ProductDetailPage`:

1. si `stock <= 0` o no es numericamente confiable, el CTA muestra `Agotado` y queda bloqueado
2. si `stock > 0` y el usuario no esta autenticado, el CTA sigue visible pero redirige a `/login`
3. si `stock > 0` y el usuario esta autenticado, el CTA permite agregar al carrito

Precedencia aplicada:

1. agotado bloquea todo
2. luego se evalua auth
3. solo entonces se permite agregar

## 6. Defensive parsing de stock

Antes:

- la UI trataba `stock` sin validacion robusta
- `undefined`, `null` o strings podian llevar a comportamiento ambiguo

Despues:

- solo se considera disponible si `Number(stock)` es finito y `> 0`
- `0`, `"0"`, `undefined`, `null`, `NaN` o string invalido se tratan como agotado
- esto aplica tanto en catalogo como en detalle

Riesgo cubierto:

- falsos positivos de stock por valores truthy o no normalizados

## 7. Compatibilidad de ID (`id` vs `_id`)

- Se detecto riesgo de ambiguedad entre `product.id` y `product._id`.
- En `ProductCard.jsx` se normalizo el uso con `product.id || product._id` para links y `data-testid`.
- En `ProductDetailPage.jsx` no fue necesario un cambio estructural adicional porque `addItem` ya usa `product.id || product._id` dentro de `CartContext`.
- No se rompio el agregado al carrito por propiedad incorrecta.
