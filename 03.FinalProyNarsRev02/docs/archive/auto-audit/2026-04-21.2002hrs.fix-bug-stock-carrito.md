# Fix de bug de stock en carrito

## 1. Resumen ejecutivo

Se aplico un fix minimo y coordinado entre backend y frontend para corregir el falso mensaje de stock agotado en la vista de carrito. El backend ahora incluye `stock` en las respuestas pobladas del carrito y el frontend dejo de interpretar la ausencia temporal del dato como `0`.

Con esto, el flujo vuelve a permitir:

- agregar productos con stock real disponible,
- incrementar `1 -> 2`,
- decrementar `2 -> 1`,
- eliminar `1 -> 0`,
- y seguir mostrando el mensaje de agotado solo cuando el stock real es `0`.

## 2. Problema corregido

Antes del fix:

- el backend del carrito devolvia `products.product` sin `stock`,
- el frontend convertia esa ausencia a `0`,
- y la validacion del carrito trataba ese `0` como agotado real.

Resultado: al presionar `+` o `-` en carrito aparecia `Este producto ya no tiene stock disponible.` aunque el producto si tuviera inventario real.

## 3. Causa raiz

La causa raiz fue una desalineacion del contrato FE/BE:

- `ecommerce-api-Nars/src/controllers/cartController.js` no incluia `stock` en `populate('products.product', ...)`.
- `ecommerce-app-Nars/src/contexts/CartContext.jsx` convertia `stock` ausente a `0`.
- `ecommerce-app-Nars/src/hooks/useCartStockValidation.js` interpretaba ese `0` como stock agotado.

## 5. Cambios aplicados

### Backend

En `cartController.js` se agrego `stock` a todas las respuestas pobladas del carrito:

```js
.populate('products.product', 'name price stock _id imagesUrl image')
```

Se aplico en:

- `getCarts`
- `getCartById`
- `getCartByUser`
- `createCart`
- `updateCart`
- `addProductToCart`

### Frontend - mapeo del carrito

En `CartContext.jsx` se reemplazo la conversion agresiva:

```js
stock: Number(product.stock || 0)
```

por una normalizacion defensiva que preserva `null` cuando el dato no viene:

```js
stock:
  product.stock == null || product.stock === ""
    ? null
    : Number.isFinite(parsedStock)
      ? parsedStock
      : null
```

### Frontend - validacion de stock

En `useCartStockValidation.js` ahora se distingue entre:

- dato faltante (`null`, `undefined`, `""`) -> no bloquear,
- dato no numerico -> no bloquear preventivamente,
- stock real `<= 0` -> bloquear con mensaje legitimo.

## 6. Justificacion tecnica de cada cambio

- Agregar `stock` en backend corrige la fuente de verdad del carrito y alinea el contrato con el resto del frontend.
- No convertir ausencia de `stock` a `0` evita falsos positivos de agotado.
- Permitir el flujo cuando `stock` falta mantiene un comportamiento defensivo seguro: el frontend no inventa stock, pero tampoco bloquea por un dato incompleto.
- Mantener el bloqueo cuando `stock = 0` preserva la regla real de negocio.
- Actualizar pruebas evita que la regresion vuelva a pasar desapercibida.

## 9. Riesgos residuales

- Si existiera algun consumidor externo del carrito que dependiera implicitamente de la proyeccion anterior, ahora recibira `stock` adicional. Es un cambio compatible, pero conviene tenerlo presente.
- La validacion frontend queda mas robusta, pero la fuente final de verdad sigue siendo backend; si a futuro aparecen respuestas parciales distintas, el frontend permitira seguir hasta que el backend responda.
- No se corrio E2E Cypress completo en esta iteracion; la validacion se hizo con pruebas unitarias/integracion acotadas al bug y al flujo de checkout backend.

## 10. Conclusion final

El bug quedo corregido con un cambio minimo y acotado. El carrito ya no marca falso agotado cuando el backend omite temporalmente `stock`, y sigue bloqueando correctamente cuando el producto realmente no tiene inventario.

El flujo central de carrito y la integracion backend con checkout permanecen estables segun la evidencia ejecutada.
