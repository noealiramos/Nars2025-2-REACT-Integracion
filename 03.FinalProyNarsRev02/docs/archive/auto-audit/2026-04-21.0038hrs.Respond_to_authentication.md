# Auditoria tecnica - II.1 Autenticacion (Registro/Login/Logout)

## 1. RESUMEN EJECUTIVO

- Estado general: **Parcial**.
- Justificacion breve: el flujo funcional de registro, login, refresh y logout si opera end-to-end contra backend real, y la app usa JWT en requests autenticados. Sin embargo, la seguridad basica queda incompleta porque `accessToken` y `refreshToken` se guardan en `localStorage`, y la restauracion de sesion del frontend confia en `userData` local sin revalidar con backend al iniciar.

## 3. FLUJO REAL (PASO A PASO)

### Registro

1. El usuario envia nombre, email y password desde `ecommerce-app-Nars/src/pages/RegisterPage.jsx:23`.
2. El frontend llama `authApi.register()` en `ecommerce-app-Nars/src/api/authApi.js:9`.
3. El backend valida con `express-validator` en `ecommerce-api-Nars/src/routes/authRoutes.js:49`.
4. `register()` en `ecommerce-api-Nars/src/controllers/authController.js:41` verifica duplicados, hashea password con bcrypt y crea el usuario.
5. El backend responde `201` con `user`, `accessToken` y `refreshToken`.
6. El frontend guarda tokens con `persistAuthSession()` en `ecommerce-app-Nars/src/services/authService.js:26`, actualiza contexto con `restoreSession()` en `ecommerce-app-Nars/src/contexts/AuthContext.jsx:50` y navega a `/`.

### Login

1. El usuario envia email y password desde `ecommerce-app-Nars/src/pages/LoginPage.jsx:22`.
2. `useAuth().login()` en `ecommerce-app-Nars/src/contexts/AuthContext.jsx:30` delega en `authService.login()`.
3. `authService.login()` llama `authApi.login()` en `ecommerce-app-Nars/src/api/authApi.js:4`.
4. `login()` en `ecommerce-api-Nars/src/controllers/authController.js:80` busca usuario, valida `hashPassword`, revisa `active`, genera JWT y refresh token, y responde `200`.
5. El frontend persiste `accessToken`, `refreshToken` y `userData` en `localStorage`, marca `isAuthenticated=true` y redirige a la ruta origen.

### Uso del token

1. `apiClient` lee `accessToken` desde `localStorage` en `ecommerce-app-Nars/src/api/apiClient.js:88`.
2. Agrega `Authorization: Bearer <token>` en cada request en `ecommerce-app-Nars/src/api/apiClient.js:90`.
3. El backend valida el Bearer token en `ecommerce-api-Nars/src/middlewares/authMiddleware.js:3`.
4. Ante `401`, el frontend intenta `POST /auth/refresh` con `refreshToken` desde `localStorage` en `ecommerce-app-Nars/src/api/apiClient.js:37`.
5. `refresh()` en `ecommerce-api-Nars/src/controllers/authController.js:125` valida existencia en DB, revoca el refresh anterior, genera nuevo par de tokens y responde `200`.
6. Si el refresh falla, `apiClient` limpia storage y dispara `auth-error` en `ecommerce-app-Nars/src/api/apiClient.js:62`, y `AuthContext` limpia el estado en `ecommerce-app-Nars/src/contexts/AuthContext.jsx:21`.

### Logout

1. El usuario hace click en `Cerrar sesion` en `ecommerce-app-Nars/src/components/organisms/SiteHeader.jsx:135`.
2. `useAuth().logout()` en `ecommerce-app-Nars/src/contexts/AuthContext.jsx:44` llama `authService.logout()`.
3. `authService.logout()` envia `POST /auth/logout` con `refreshToken` si existe en `ecommerce-app-Nars/src/services/authService.js:34`.
4. El backend revoca el refresh token con `RefreshToken.updateMany()` en `ecommerce-api-Nars/src/controllers/authController.js:171`.
5. El frontend siempre elimina `accessToken`, `refreshToken` y `userData` en el bloque `finally` de `ecommerce-app-Nars/src/services/authService.js:43` y pone `user=null`, `isAuthenticated=false` en `ecommerce-app-Nars/src/contexts/AuthContext.jsx:46`.

// ecommerce-api-Nars/src/middlewares/authMiddleware.js:3
const authMiddleware = (req, res, next) => {
  const auth = req.headers['authorization'] || '';
  const parts = auth.split(' ');
  const isBearer = parts.length === 2 && /^Bearer$/i.test(parts[0]);
  const token = isBearer ? parts[1] : null;

  if (!token) return res.status(401).json({ message: 'Unauthorized: missing Bearer token' });

  jwt.verify(token, process.env.JWT_SECRET, (err, decoded) => {
    if (err?.name === 'TokenExpiredError') return res.status(401).json({ message: 'Token expired' });
    if (err) return res.status(403).json({ message: 'Invalid token' });
    if (decoded?.userId && !decoded.id) decoded.id = decoded.userId;
    req.user = decoded;
    next();
  });
};

export function persistAuthSession(data) {
  localStorage.setItem(STORAGE_KEYS.accessToken, data.accessToken);
  localStorage.setItem(STORAGE_KEYS.refreshToken, data.refreshToken);
  localStorage.setItem(STORAGE_KEYS.userData, JSON.stringify(data.user));
}

// ecommerce-app-Nars/src/services/authService.js:32
export async function logout() {
  try {
    const refreshToken = localStorage.getItem(STORAGE_KEYS.refreshToken);
    if (refreshToken) {
      await authApi.logout(refreshToken);
    }
  } finally {
    localStorage.removeItem(STORAGE_KEYS.accessToken);
    localStorage.removeItem(STORAGE_KEYS.refreshToken);
    localStorage.removeItem(STORAGE_KEYS.userData);
  }
}

## 6. RIESGOS DETECTADOS

### Seguridad

- `refreshToken` y `accessToken` se guardan en `localStorage` en `ecommerce-app-Nars/src/services/authService.js:26`; esto expone la sesion a robo por XSS.
- `AuthContext` restaura sesion solo con `userData` local en `ecommerce-app-Nars/src/contexts/AuthContext.jsx:12`; un `userData` stale o manipulado puede pintar UI autenticada antes de una validacion real.
- El logout del contexto en `ecommerce-app-Nars/src/contexts/AuthContext.jsx:44` no espera `await authLogout()`. Funcionalmente limpia estado, pero la UI no conoce el resultado del request de revocacion.

### UX

- Cuando falla refresh, el usuario es expulsado al login por evento global `auth-error`; no hay mensaje dedicado que explique expiracion o revocacion de sesion.
- El warning inicial de Cypress sobre verificacion de `baseUrl` muestra un comportamiento de arranque algo fragil del frontend de pruebas, aunque luego la suite pasa completa.

### Consistencia

- Registro usa `authApi.register()` directo en `ecommerce-app-Nars/src/pages/RegisterPage.jsx:35`, mientras login usa `useAuth().login()` y `authService`. La autenticacion no sigue una unica via de orquestacion.
- El header observado `ratelimit-policy: 100;w=60` aparece junto con `ratelimit-remaining: 999` en requests auth; esto sugiere mezcla de cabeceras entre limitador global y limitador auth, y merece revision de consistencia operativa.

## 7. BRECHAS (GAPS)

- Mover refresh token fuera de `localStorage` hacia cookie `HttpOnly`, `Secure`, `SameSite`.
- Rehidratar sesion consultando backend (`/users/me`) o validando token al iniciar, en lugar de confiar solo en `userData` local.
- Unificar registro y login dentro de la misma capa (`AuthContext`/`authService`) para reducir divergencias de flujo.
- Exponer mejor feedback de expiracion/revocacion de sesion al usuario final.
- Agregar evidencia automatizada explicita que valide `localStorage` tras logout normal por click, no solo en el flujo de refresh revocado.

## 8. PLAN DE MEJORA (SIN IMPLEMENTAR)

### Lista priorizada

1. Migrar refresh token a cookie `HttpOnly` y dejar el access token con estrategia mas corta o memoria volatile.
2. Agregar bootstrap auth con `GET /users/me` para revalidar sesion al cargar la app.
3. Centralizar registro/login/logout/restore en `AuthContext` + `authService`.
4. Agregar prueba E2E dedicada a logout normal validando limpieza de `accessToken`, `refreshToken` y `userData` despues del click.
5. Revisar encabezados de rate limiting para que el set observado sea coherente y auditable.

### Bajo riesgo (quick wins)

- Unificar el flujo de registro con `AuthContext`.
- Mostrar mensaje de sesion expirada o revocada cuando ocurra `auth-error`.
- Crear prueba E2E especifica para logout por UI con verificacion de storage.

### Medio / alto riesgo

- Redisenar almacenamiento de tokens con cookies seguras y cambios coordinados backend/frontend.
- Revalidar sesion en arranque con cambios de UX para estados de carga, expiracion y fallback offline.
