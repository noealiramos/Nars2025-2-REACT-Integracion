# Auditoría técnica: cart loading issue

Fecha: `2026-04-22 12:06`

## Resumen ejecutivo

La causa raíz del parpadeo del carrito no es un `refetch` extra con `GET /cart/user` después de cada click en `+/-`.

El problema proviene de que `updateQuantity` en `CartContext` activa un `loading` global (`LOAD_START`), y `CartPage` usa ese flag para ocultar toda la UI del carrito y mostrar `Cargando carrito...`. Eso desmonta visualmente la lista y el resumen, y luego los vuelve a montar cuando llega la respuesta del backend.

Sí existe una resincronización completa del carrito en cada cambio, pero ocurre vía `PUT /cart/:id` enviando el arreglo completo de productos y reemplazando luego todo el estado del carrito con la respuesta completa del backend. No hay un `GET` adicional disparado por un `useEffect` tras cada actualización.

## Hallazgos

### 1. ¿Qué función se ejecuta al hacer click en `+` o `-`?

1. `CartItem` ejecuta `onChangeQuantity(item.id, item.quantity - 1)` o `onChangeQuantity(item.id, item.quantity + 1)`.
   Referencia: `ecommerce-app-Nars/src/components/molecules/CartItem.jsx:45-57`
2. `CartPage` pasa `updateQuantity` del contexto a cada `CartItem`.
   Referencia: `ecommerce-app-Nars/src/pages/CartPage.jsx:64-70`
3. La función real ejecutada es `updateQuantity(productId, quantity)` dentro de `CartContext`.
   Referencia: `ecommerce-app-Nars/src/contexts/CartContext.jsx:230-266`

### 2. ¿Después de actualizar cantidad se llama nuevamente a obtener todo el carrito?

No mediante `cartApi.getCurrent()`.

Hallazgo exacto:

1. `cartApi.getCurrent()` solo se llama dentro del `useEffect` de carga inicial/autenticación.
   Referencia: `ecommerce-app-Nars/src/contexts/CartContext.jsx:96-125`
2. Ese `useEffect` depende solo de `isAuthenticated`.
   Referencia: `ecommerce-app-Nars/src/contexts/CartContext.jsx:125`
3. Tras `+/-`, `updateQuantity` llama `persistCartItems(nextItems)`.
   Referencia: `ecommerce-app-Nars/src/contexts/CartContext.jsx:252-259`
4. Si ya existe `cartId`, `persistCartItems` ejecuta `cartApi.update(state.cartId, productsPayload)`.
   Referencia: `ecommerce-app-Nars/src/contexts/CartContext.jsx:152-158`
5. `cartApi.update` hace `PUT /cart/:id` con el arreglo completo `products`.
   Referencia: `ecommerce-app-Nars/src/api/cartApi.js:15-18`

Conclusión: no hay `refetch` completo por `GET`, pero sí hay una actualización completa del carrito por `PUT` y un reemplazo completo del estado frontend con la respuesta del backend.

### 3. ¿Existe un loading global del carrito?

Sí.

`CartContext` tiene un solo flag global: `isLoading`.
Referencia: `ecommerce-app-Nars/src/contexts/CartContext.jsx:9-14`

Ese flag se activa en:

1. carga inicial del carrito: `LOAD_START` en `loadCart()`.
   Referencia: `ecommerce-app-Nars/src/contexts/CartContext.jsx:99-101`
2. agregar producto: `addItem()`.
   Referencia: `ecommerce-app-Nars/src/contexts/CartContext.jsx:179`
3. eliminar producto: `removeItem()`.
   Referencia: `ecommerce-app-Nars/src/contexts/CartContext.jsx:200`
4. vaciar carrito: `clearCart()`.
   Referencia: `ecommerce-app-Nars/src/contexts/CartContext.jsx:218`
5. actualizar cantidad: `updateQuantity()`.
   Referencia: `ecommerce-app-Nars/src/contexts/CartContext.jsx:256`

No existe un `loading` granular por item.

### 4. ¿Qué dispara el mensaje `Cargando carrito...`?

`CartPage` renderiza ese mensaje cuando `isLoading === true`.
Referencia: `ecommerce-app-Nars/src/pages/CartPage.jsx:24-26`

Como `updateQuantity()` dispara `LOAD_START`, cada click en `+/-` vuelve `isLoading` a `true`, aunque el usuario no esté entrando a la página sino solo mutando un item.

### 5. ¿Se está usando `useEffect` que vuelve a pedir el carrito tras cada cambio?

No.

El único `useEffect` del `CartContext` depende de `isAuthenticated`, no de `items`, `cartId`, `totalPrice` ni de la cantidad de productos.
Referencia: `ecommerce-app-Nars/src/contexts/CartContext.jsx:96-125`

No se encontró ningún `useEffect` que vuelva a ejecutar `cartApi.getCurrent()` después de cada mutación del carrito.

### 6. ¿Se está re-renderizando todo el componente por cambio de estado global?

Sí.

Hay dos niveles del problema:

1. `CartPage` depende de `isLoading` global y hace render condicional de toda la sección de items y del resumen. Cuando `isLoading` pasa a `true`, desaparece la lista completa.
   Referencias: `ecommerce-app-Nars/src/pages/CartPage.jsx:24-25`, `:43-79`
2. `CartContext` reemplaza `items` completos con `SET_CART` y entrega un `value` nuevo del contexto en cada render, por lo que los consumidores de `useCart()` se vuelven a renderizar.
   Referencias: `ecommerce-app-Nars/src/contexts/CartContext.jsx:54-61`, `:278-290`

Adicionalmente, `SiteHeader` también consume `useCart()` por `totalItems`, así que también participa del rerender del contexto.
Referencia: `ecommerce-app-Nars/src/components/organisms/SiteHeader.jsx:8-10`

## Flujo actual

### Flujo paso a paso desde click hasta render

1. El usuario hace click en `-` o `+` en `CartItem`.
2. `CartItem` invoca `onChangeQuantity(...)`.
3. `CartPage` delega a `updateQuantity(...)` del `CartContext`.
4. `updateQuantity` valida stock y calcula `nextItems`.
5. `updateQuantity` hace `dispatch({ type: "LOAD_START" })`.
6. El reducer pone `isLoading: true`.
7. `CartPage` se vuelve a renderizar y entra a la rama que muestra `Cargando carrito...`.
8. Como la lista y el resumen dependen de `!isLoading`, ambos dejan de renderizarse.
9. `persistCartItems(nextItems)` arma `productsPayload` con todo el carrito.
10. El frontend hace `PUT /cart/:id` con todo el arreglo `products`.
11. El backend reemplaza `products` completos del carrito con `$set: { products }`.
    Referencia: `ecommerce-api-Nars/src/controllers/cartController.js:115-118`
12. El backend responde con el carrito completo poblado.
    Referencia: `ecommerce-api-Nars/src/controllers/cartController.js:115-124`
13. `applyCartResponse(updated)` dispara `SET_CART`.
14. El reducer actualiza `items`, `cartId`, `error` e `isLoading: false`.
15. `CartPage` vuelve a renderizar y muestra otra vez toda la UI del carrito.

## Problema identificado

### Causa raíz

La causa raíz es la combinación de estos tres puntos:

1. `updateQuantity()` usa un `loading` global de página en lugar de un estado de mutación granular por item.
2. `CartPage` usa ese `loading` global para ocultar completamente la lista del carrito y el resumen.
3. El flujo de persistencia reemplaza el estado completo del carrito con la respuesta completa del backend en cada cambio.

### Qué parte provoca el parpadeo o recarga visual

La pieza que provoca el efecto de “recarga” visible es principalmente esta condición de render:

1. `isLoading && <p className="cart-page__status">Cargando carrito...</p>`
2. `!isLoading && !error && !isGuestMode && !isEmpty && (...)`

Referencia: `ecommerce-app-Nars/src/pages/CartPage.jsx:24-26` y `:54-79`

En cuanto `LOAD_START` corre desde `updateQuantity`, el árbol visual del carrito se desmonta y luego se monta otra vez. Ese desmontaje/remontaje es el parpadeo que el usuario percibe como “se recarga el carrito”.

## Diagnóstico técnico

### Frontend

1. `CartContext` trata la carga inicial y las mutaciones del carrito como si fueran el mismo estado de loading.
2. `updateQuantity`, `removeItem`, `clearCart` y `addItem` activan el mismo `isLoading` global.
3. `CartPage` no diferencia entre “estoy cargando el carrito inicial” y “estoy sincronizando un solo item”.
4. No hay actualización optimista; la UI espera a la respuesta del backend para volver a pintar el carrito.

### Backend

1. El backend no hace un segundo `GET` tras la actualización.
2. El endpoint `PUT /cart/:id` reemplaza el arreglo completo `products`.
3. La respuesta vuelve con el carrito completo poblado.

Esto es consistente con los contratos actuales, pero empuja al frontend a resincronizar todo el estado aunque solo cambie un item.

## Plan propuesto (sin ejecutar)

Objetivo: eliminar el parpadeo sin romper contratos frontend/backend ni cambiar endpoints.

### Archivo 1: `ecommerce-app-Nars/src/contexts/CartContext.jsx`

Cambios propuestos:

1. Separar el loading inicial del carrito del loading de mutación.
   Propuesta mínima: mantener `isLoading` solo para bootstrap inicial y agregar `pendingItemIds` o `itemLoadingMap` para mutaciones por item.
2. No disparar `LOAD_START` global dentro de `updateQuantity()`.

## Riesgos

### ¿Hay riesgo de desincronización con backend?

Sí, pero es controlable.

Riesgo:

1. una actualización optimista puede mostrar una cantidad temporal que luego backend rechace o ajuste por stock

Mitigación:

1. conservar validación previa con `useCartStockValidation`
2. reconciliar siempre con la respuesta del backend al terminar
3. hacer rollback si la petición falla
4. mantener mensajes de error actuales

## Conclusión

El problema no es un `refetch` redundante por `GET`, sino un manejo de `loading` demasiado global para una mutación pequeña. Cada `+/-` activa el mismo estado de carga usado para el bootstrap inicial, y `CartPage` responde ocultando toda la UI del carrito. Ese diseño, combinado con la resincronización completa del estado desde la respuesta del `PUT`, genera el efecto visual de recarga.

La corrección recomendada es separar `loading` inicial de `loading` por item, mantener la lista visible durante la mutación y reconciliar optimistamente con backend sin tocar contratos actuales.
