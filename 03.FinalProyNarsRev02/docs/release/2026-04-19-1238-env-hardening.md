- Riesgo alto: el frontend depende de un fallback hardcodeado a `http://localhost:3000/api`, mientras la documentacion operativa y los `.env.local` usan `3001`; esto favorece configuraciones inconsistentes.

- Riesgo medio: Cypress usa URLs embebidas en multiples archivos y no centralizadas por entorno.
- Riesgo medio: `server.js` y varios documentos asumen localhost para visibilidad operativa; para produccion conviene derivar URL publica o registrar solo puerto/entorno.

### Plan de cambios propuesto
1. Crear `ecommerce-api-Nars/.env.example` y `ecommerce-app-Nars/.env.example` con placeholders seguros y comentarios minimos de uso.
2. Eliminar fallbacks productivos hardcodeados en runtime del frontend; requerir `VITE_API_URL` o definir una estrategia controlada por modo/entorno.

6. Actualizar README raiz y README frontend/backend para explicar variables requeridas, ejemplos locales y consideraciones de Render + Atlas.
7. Mantener scripts/tests de performance con variables parametrizadas y dejar localhost solo como fallback explicitamente etiquetado para desarrollo/test, no para produccion.

- Priorizar cambios no funcionales: plantillas `.env.example`, documentacion y parametrizacion.
