Proyecto: ecommerce-api-Nars

Actúa como un Senior Backend Security Engineer especializado en Node.js, Express, MongoDB, Mongoose y hardening de APIs REST para producción.

Tu objetivo es corregir las vulnerabilidades detectadas por auditoría y dejar el backend en un estado seguro y estable antes del despliegue a producción, sin romper la funcionalidad actual ni los tests existentes.

Contexto del proyecto:
- Backend Node.js + Express 5
- MongoDB + Mongoose
- JWT con accessToken y refreshToken
- Winston logger
- Swagger/OpenAPI
- Existen pruebas automatizadas que actualmente pasan
- Se detectaron vulnerabilidades en dependencias mediante npm audit, incluyendo validator y minimatch
- Se recomendó elevar el costo de bcrypt de 10 a 12
- El proyecto ya cuenta con rate limiting, sanitización NoSQL, RBAC, ownerOrAdmin, refresh tokens y logs de seguridad

Tareas que debes ejecutar:

1. Auditar dependencias
- Ejecuta un análisis completo equivalente a npm audit
- Identifica dependencias vulnerables directas e indirectas
- Explica cuáles son realmente explotables en este proyecto y cuáles no son bloqueantes

2. Corregir vulnerabilidades
- Aplica las actualizaciones mínimas y más seguras posibles
- Prioriza npm audit fix antes de cambios disruptivos
- Usa npm audit fix --force solo si es estrictamente necesario
- Si una actualización mayor rompe compatibilidad, propón alternativa concreta
- Actualiza package.json y package-lock.json si corresponde

3. Endurecer autenticación
- Revisa la configuración de bcrypt
- Incrementa saltRounds de 10 a 12 en la parte adecuada del proyecto
- Verifica que el cambio no rompa login, registro ni pruebas existentes
- Si existe riesgo de impacto en rendimiento, documéntalo con claridad

4. Validar estabilidad
- Ejecuta o simula verificación de:
  - tests unitarios
  - tests de integración
  - flujo de login
  - flujo de refresh token
  - endpoints críticos
- Si algo falla, corrígelo
- No dejes cambios a medias

5. Revisar seguridad complementaria
- Verifica que .env, .envProduction u otros secretos no estén siendo rastreados
- Confirma que .gitignore cubre archivos sensibles
- Revisa que no haya credenciales hardcodeadas
- Verifica que el logging no exponga tokens o passwords
- Revisa que Swagger no documente secretos reales ni ejemplos inseguros

6. Generar salida útil y accionable
Entrega un reporte final en Markdown con esta estructura:

# Reporte de Remediación de Vulnerabilidades
## 1. Vulnerabilidades encontradas
## 2. Cambios realizados
## 3. Archivos modificados
## 4. Riesgos que permanecen
## 5. Resultado de pruebas
## 6. Veredicto final de preparación para producción

Además:
- Muestra el diff o fragmentos exactos del código modificado
- Indica con precisión qué comando ejecutaste o debió ejecutarse
- Explica si fue necesario usar npm audit fix o npm audit fix --force
- Indica si el backend queda listo para producción o si aún hay bloqueantes

Restricciones importantes:
- No hagas refactors innecesarios
- No cambies arquitectura si no es indispensable
- No elimines funcionalidad existente
- No rompas compatibilidad con el frontend futuro
- Minimiza deuda técnica
- Prioriza cambios seguros, trazables y reversibles

Criterio de éxito:
- Vulnerabilidades críticas y altas mitigadas o justificadas
- bcrypt endurecido
- tests pasando
- backend estable
- proyecto listo para seguir con el desarrollo del frontend sin arrastrar deuda de seguridad

Evita recomendaciones genéricas. Trabaja sobre el código real del proyecto y entrega acciones concretas aplicables.
